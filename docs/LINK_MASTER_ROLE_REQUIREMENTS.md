# Link Master Role Requirements & SSH Self-Authentication

## Critical Constraints

### 1. Every Device Must Have At Least One Master Link

**Requirement**: Every device in the topology MUST be assigned as `node_a` (MASTER) on at least one link. You **CANNOT** have a device with only slave links (node_b).

**Reason**: 
- Master links are responsible for key generation and rotation
- The device with role `node_a` on a link generates the initial CAK and rotates keys
- A device with zero master links has no responsibility to generate keys → no key rotation → keys become stale
- In MKA peer-to-peer model, the master generates, slave receives

**Evidence - Bug Case (v3.3.0)**:
- MX6 was configured as `node_b` (SLAVE) on ALL 4 of its links
- Result: Zero master_links for MX6 → no key generation responsibility
- Symptom: 60+ stale keys accumulating on all links where MX6 is peer
- Fix: Reassigned MX6 to `node_a` (MASTER) on all 4 links
- Outcome: MX6 now generates keys with "PENDING KEY PROMOTED" events visible in logs

**Implementation Validation**:
- `qkd_onbox.py` line 2849: `master_links = [link for link in links if link.get("role") == "master"]`
- If `master_links` is empty → device generates zero keys
- Orchestrator should validate at deploy time that each device has `len(master_links) >= 1`

### 2. SSH Self-Authentication Required in authorized_keys

**Requirement**: Each device's public key (e.g., `sae-002_id_ed25519.pub`) MUST be present in its own `authorized_keys` file for the macsec_user account.

**Reason**:
- `qkd_onbox.py` runs every 60 seconds on each device
- During key rotation, it needs to SSH to peer devices to read their MKA state
- Example: MX2 reads MX1's peer-view status via `ssh -i sae-002_id_ed25519 macsec_user@MX1_IP "show mka statistics interface et-0/0/0"`
- BUT: MX2 also needs to SSH to **itself** (localhost/127.0.0.1 or via management IP) in peer topology queries
- Without self-key in authorized_keys → SSH fails with "Permission denied (publickey,password,keyboard-interactive)"
- Failed SSH → JSON state files not updated → cascading health monitor failures

**Evidence - Bug Case (v3.3.1 early)**:
- MX1 had `sae-001_id_ed25519.pub` in its authorized_keys ✓
- MX2 was **missing** `sae-002_id_ed25519.pub` from its own authorized_keys ✗
- Pattern repeated across all devices
- Root cause: `install_peer_authorized_keys()` in `lib/qkd/identity.py` only synced **peer** keys from `linked_peer_sources(device)`
- Did NOT include the device itself as a source

**Fix Applied (commit acc3bb6)**:
```python
# lib/qkd/identity.py line 931
source_names = linked_peer_sources(device)
source_names.add(target)  # Always include device itself so it can SSH to itself
```

**Implementation Details**:
- Junos path: `system login user macsec_user authentication ssh-ed25519 "<public_key_material>"`
- Each device's qkd_peer_cmd_ed25519.pub is generated during `create` phase
- Deployed to authorized_keys during `deploy` phase via `install_peer_authorized_keys()`
- Must persist through weekly key regeneration cycles (currently regenerated by separate process)

## Topology Design Validation Checklist

When designing a new topology in `config/inventory/input/`:

- [ ] **Master Link Distribution**: Verify every device (MX1-MX6, ACX1-ACX5) appears as `node_a` on at least one link
- [ ] **Self-Authentication**: Verify `install_peer_authorized_keys()` includes device itself (source_names.add(target))
- [ ] **Peer Reachability**: Verify all linked peer names exist in the device inventory
- [ ] **Link Count**: Ensure topology has correct total link count (typically 10 primary + N extra)

## Example: Correct Topology Pattern

```yaml
links:
  - id: MX1-MX2
    node_a: MX1      # MX1 is MASTER on this link
    interface_a: et-0/0/0
    node_b: MX2      # MX2 is SLAVE on this link
    interface_b: et-0/0/0
    
  - id: MX2-MX3
    node_a: MX2      # MX2 is MASTER on this link (different from above)
    interface_a: et-0/0/1
    node_b: MX3
    interface_b: et-0/0/0

  # Result: 
  # MX1 has 1 master link (MX1-MX2) ✓
  # MX2 has 2 master links (MX1-MX2 as slave, MX2-MX3 as master) ✓
  # MX3 has at least this 1 master link ✓
```

## Example: INCORRECT Topology Pattern (Anti-Pattern)

```yaml
links:
  - id: MX1-MX2
    node_a: MX1
    node_b: MX2

  - id: MX2-MX3
    node_a: MX1      # MX1 master again
    node_b: MX3

  - id: MX3-MX4
    node_a: MX1      # MX1 master again
    node_b: MX4

  # Result:
  # MX1 has 3 master links ✓
  # MX2 has 0 master links ✗ FAIL - only appears as node_b
  # MX3 has 0 master links ✗ FAIL - only appears as node_b
  # MX4 has 0 master links ✗ FAIL - only appears as node_b
```

## Orchestration Impact

- **`create` phase**: Generates role assignments based on `node_a`/`node_b` in topology
- **`deploy` phase**: 
  - Bootstrap: Creates SCRIPT_USER on devices
  - Peer SSH sync: Calls `install_peer_authorized_keys()` with device self-authentication included
  - qkd_onbox deploy: Distributes rotation script to each device
  - Config push: Applies Junos MACsec configuration
- **`validate` phase**: Should verify device has `len(master_links) >= 1` before marking healthy
