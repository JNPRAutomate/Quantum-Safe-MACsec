# KME Orchestrator Architecture

## Purpose

`kme_orchestrator.py` manages the KME infrastructure lifecycle on the remote host:

- host bootstrap and SSH access
- remote dependency installation
- ETSI repository preparation
- compose generation/deployment
- certificate installation (consuming QKD PKI output)
- KME restart/validate/status/destroy workflows

It does **not** generate PKI.

## Scope owned by KME orchestrator

- remote host lifecycle for Ubuntu/RHEL
- Docker and compose runtime readiness
- local KME image build (`etsi-kme:local`)
- dynamic compose generation (`lib/kme/compose.py`)
- shared PostgreSQL + per-KME container model
- state tracking in `config/kme/state/<env>-state.yaml`

Entrypoint: `kme_orchestrator.py`

## Module map (`lib/kme`)

- `bootstrap.py` - SSH key/alias/remote workspace initialization
- `install_host.py` - OS dependency installation and verification
- `build_env.py` - repo sync, compose generation/upload, network prep, optional compose up
- `build_image.py` - cargo and docker image build on remote host
- `cert_install.py` - cert staging, SAN checks, remote cert upload
- `db_init.py` - PostgreSQL schema initialization
- `deploy.py` - compose up for selected services
- `restart.py` - KME-only restart with DB safety
- `status.py` - read-only deployment visibility
- `validate.py` - runtime checks across SSH/docker/compose/certs
- `destroy.py`, `stop.py`, `start.py` - lifecycle controls
- `compose.py` - dynamic container/service generation
- `state.py` - per-environment state persistence

## Create workflow

`create` orchestrates:

1. `bootstrap`
2. `install-host`
3. `build-env` (no_up=True in create flow)
4. `build-image`
5. `install-certs`
6. `db-init`
7. `deploy`
8. optional `validate`

## Runtime model and naming

- one shared DB service: `qkd-postgres`
- DB container: `{owner}-qkd-postgres`
- KME services: `kme01..kmeNN`
- KME containers: `{owner}-kme01..{owner}-kmeNN`
- image: `etsi-kme:local`
- network: external `qkd_net` (driver/subnet/gateway from config)

## Restart policy

Normal restart is KME-only:

- restarts selected KME containers
- does not restart/remove PostgreSQL
- does not perform compose down/up cycles

This protects DB persistence during cert refresh operations.

## Configuration and state

Configuration:

- `config/kme/lab.yaml`
- `config/kme/live.yaml`

State:

- `config/kme/state/<environment>-state.yaml`

State includes step completion flags and metadata for:

- bootstrap
- install_host
- build_env
- build_image
- cert_install
- restart
- validate/status-related context

## Integration boundary with QKD orchestrator

KME cert installation reads the active runtime PKI profile from:

- `config/runtime/pki_profile.yaml`

and consumes cert/trust material generated by QKD workflow under `certs/`.
