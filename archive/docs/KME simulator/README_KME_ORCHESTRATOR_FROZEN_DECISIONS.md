# KME Orchestrator Architecture — Frozen Decisions

> Status: **Frozen baseline for current KME orchestrator refactor**
>
> These decisions are treated as accepted unless explicitly changed later.

---

## 1. Ownership boundaries

### qkd_orchestrator.py

`qkd_orchestrator.py` remains the only component responsible for:

- QKD topology and runtime inventory
- Juniper device artifacts
- PKI generation
- self-signed profile generation
- hierarchical CA profile generation
- SAE certificates
- KME certificates
- trust bundles
- Juniper deployment and validation

`qkd_orchestrator.py` produces:

```text
config/runtime/*
certs/*
```

### kme_orchestrator.py

`kme_orchestrator.py` is a KME infrastructure lifecycle manager.

It consumes PKI material generated by `qkd_orchestrator.py` and manages:

- SSH bootstrap to remote KME host
- Ubuntu/RHEL host installation
- Docker installation and validation
- Rust/toolchain validation and build
- ETSI GS QKD 014 repository clone/update
- local Docker image build
- Docker network creation/validation
- Docker compose deployment
- certificate installation into ETSI repo `certs/`
- KME-only restart
- status checks
- validation checks

`kme_orchestrator.py` must **never generate PKI material**.

The only intentional intersection point is:

```text
qkd_orchestrator.py -> certs/* -> kme_orchestrator.py
```

## 2. KME-to-Juniper model

Accepted model:

```text
1 Juniper device = 1 KME container
```

Examples:

```text
pair with 2 Juniper devices   -> kme01, kme02
ring with 5 Juniper devices   -> kme01..kme05
chain with 10 Juniper devices -> kme01..kme10
```

The compose file can define a fixed maximum set of KME containers, currently up to 11:

```text
kme01..kme11
```

The active topology determines how many KME containers are used.

---

## 3. PostgreSQL model

Accepted model:

```text
single shared PostgreSQL container
```

Service name:

```text
qkd-postgres
```

Container name pattern:

```text
<user>-qkd-postgres
```

Example:

```text
aterren-qkd-postgres
```

The database is shared by all KME containers.

Deprecated models:

```text
postgres-kme1
postgres-kme2
postgres-kme
```

These belong to older lab designs and must not be used as the new default.

---

## 4. Restart policy

Normal restart must restart KME containers only.

Allowed:

```bash
docker restart <user>-kme01 <user>-kme02 ... <user>-kmeNN
```

Forbidden during normal restart:

```bash
docker restart <user>-qkd-postgres
```

Forbidden during normal restart:

```bash
docker compose down -v
```

Forbidden during normal restart:

```bash
docker compose up -d
```

Reason:

- KME certificates are mounted into KME containers from `./certs:/certs:ro`.
- KME containers only need a restart to reload certificates.
- PostgreSQL persistence must survive normal KME maintenance.
- Restart must not recreate networks, volumes, or DB containers.

---

## 5. Docker image naming

Accepted Docker image name:

```text
etsi-kme:local
```

Rejected image name:

```text
aterren-etsi-kme:local
```

Reason:

- The Docker image is generic.
- The Docker image should not include a user name.
- User identity belongs in container names, not image names.

---

## 6. Docker compose naming and location

Accepted compose file name:

```text
docker-compose-kme.yml
```

Accepted location in QKD repository:

```text
config/kme/compose/docker-compose-kme.yml
```

Reason:

- There is one standard KME compose topology.
- The compose defines the maximum supported KME layout.
- Runtime topology determines active KME count.
- The compose is infrastructure-as-code and must be versioned with the QKD project.
- The compose is copied into the ETSI repository when needed.

Rejected name:

```text
docker-compose-prod.yml
```

Reason:

- The compose is not specifically “prod”; it is the standard KME compose.

---

## 7. Docker service and container naming

### Service names

Service names are generic and do not contain the user name:

```text
qkd-postgres
kme01
kme02
...
kme11
```

### Container names

Container names include the user prefix to avoid collisions on shared hosts:

```text
<user>-qkd-postgres
<user>-kme01
<user>-kme02
...
<user>-kme11
```

Example:

```text
aterren-qkd-postgres
aterren-kme01
aterren-kme02
...
aterren-kme11
```

## 8. Network model

Network driver must be configurable.

Supported drivers:

```text
ipvlan
macvlan
```

Default driver:

```text
ipvlan
```

Reason:

- Historical VQFX labs used macvlan.
- Newer Linux/RHEL/Ubuntu KME host designs use ipvlan.
- The orchestrator must not hardcode only one network type.

### Current lab defaults

Subnet:

```text
100.100.100.0/24
```

Gateway / core switch:

```text
100.100.100.50
```

KME host IP:

```text
100.100.100.254
```

Database IP:

```text
100.100.100.30
```

KME container IP range:

```text
kme01 -> 100.100.100.10
kme02 -> 100.100.100.11
...
kme11 -> 100.100.100.20
```

### IPVLAN create example

```bash
docker network create \
  -d ipvlan \
  --subnet=100.100.100.0/24 \
  --gateway=100.100.100.50 \
  -o parent=<parent-interface> \
  -o ipvlan_mode=l2 \
  qkd_net
```

### MACVLAN create example

```bash
docker network create \
  -d macvlan \
  --subnet=100.100.100.0/24 \
  --gateway=100.100.100.50 \
  -o parent=<parent-interface> \
  qkd_net
```

## 10. Bootstrap scope

Bootstrap must run before host installation.

Reason:

- The local workstation must first establish stable SSH access to the remote KME server.
- All later phases depend on SSH access.

Bootstrap responsibilities:

1. Generate local SSH key pair if missing.
2. Upload public key to the remote Linux user.
3. Configure local SSH alias.
4. Verify passwordless SSH.
5. Detect remote OS family.
6. Create remote workspace under the remote login user.
7. Clone or update the ETSI GS QKD 014 repository.

Bootstrap must not:

- install Docker
- install Rust
- build Docker images
- create Docker networks
- create containers
- install certificates
- restart containers

---

## 11. Install-host scope

`install-host` runs after bootstrap.

It installs host prerequisites on the remote Ubuntu/RHEL server.

Ubuntu responsibilities:

- install package prerequisites
- install Docker CE / compose plugin
- enable and start Docker
- add remote user to Docker group
- verify Docker and compose

RHEL responsibilities:

- install package prerequisites using `dnf`
- add Docker CE repo
- install Docker CE / compose plugin
- enable and start Docker
- add remote user to Docker group
- verify Docker and compose

---

## 12. Build-image scope

`build-image` runs after install-host and bootstrap.

Responsibilities:

- use cloned ETSI repository
- install Rust if missing
- source Cargo environment
- set `DATABASE_URL`
- set `SQLX_OFFLINE=true`
- run `cargo build --release`
- build local Docker image:

```bash
docker build --no-cache -t etsi-kme:local .
```

The image is built locally and reused by compose.

---

## 13. Build-env scope

`build-env` prepares Docker runtime infrastructure.

Responsibilities:

- copy `config/kme/compose/docker-compose-kme.yml` to the ETSI repo
- ensure Docker network exists
- start `qkd-postgres` if required
- start KME compose services if required

`build-env` may use compose, but normal `restart` must not use compose down/up.

---

## 14. Install-certs scope

`install-certs` consumes certificates generated by `qkd_orchestrator.py`.

Responsibilities:

- read runtime PKI profile
- collect self-signed or hierarchical KME certificate material
- validate KME certificate SAN IP presence
- copy material into remote ETSI repo `certs/`
- create `root.crt` alias for KME runtime compatibility

`install-certs` must not generate certificates.

---

## 15. Lifecycle command order

Accepted lifecycle:

```text
bootstrap
  -> install-host
  -> build-image
  -> build-env
  -> install-certs
  -> restart
  -> validate
  -> status
```

## 16. Commands targeted for kme_orchestrator.py

```text
show-config
bootstrap
install-host
build-image
build-env
install-certs
restart
db-init
status
validate
```

`db-init` targets only the shared DB container:

```text
<user>-qkd-postgres
```

It must not initialize multiple DB containers.

## 17. Decisions frozen in this baseline

These are frozen as of this baseline:

- KME orchestrator consumes PKI only.
- QKD orchestrator generates PKI only.
- 1 Juniper device maps to 1 KME container.
- One shared DB container is used for all KME containers.
- Shared DB service name is `qkd-postgres`.
- Shared DB container name is `<user>-qkd-postgres`.
- KME services are `kme01..kme11`.
- KME containers are `<user>-kme01..kme11`.
- Docker image is `etsi-kme:local`.
- Compose file is `docker-compose-kme.yml`.
- Compose file lives under `config/kme/compose/`.
- Network driver is configurable.
- Default driver is ipvlan.
- Current lab subnet is `100.100.100.0/24`.
- Current lab gateway is `100.100.100.50`.
- Current lab host IP is `100.100.100.254`.
- Current lab DB IP is `100.100.100.30`.
- Current lab KME IP range is `100.100.100.10-100.100.100.20`.
- Bootstrap runs before install-host.
- Normal restart never touches the DB.
- Normal restart never runs `docker compose down -v`.
