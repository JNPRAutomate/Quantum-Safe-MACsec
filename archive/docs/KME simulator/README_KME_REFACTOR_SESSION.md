# KME Orchestrator Refactor README

## Scope

This document captures the architecture and implementation decisions from the KME orchestrator refactor session.

The goal is to evolve `kme_orchestrator.py` from a certificate copy helper into a lifecycle manager for the KME host infrastructure.

The KME orchestrator manages the KME-side environment only. It does not generate PKI material.

---

## Ownership Boundaries

### qkd_orchestrator.py

`qkd_orchestrator.py` remains responsible for:

- QKD topology generation
- Juniper runtime inventory
- PKI generation
- self-signed PKI generation
- hierarchical CA PKI generation
- SAE certificates
- KME certificates
- trust bundles
- Juniper provisioning
- Juniper validation
- Juniper cleanup

It produces:

```text
config/runtime/*
certs/*
```

### kme_orchestrator.py

`kme_orchestrator.py` consumes PKI material already generated by `qkd_orchestrator.py`.

It is responsible for:

- remote KME host bootstrap
- SSH key setup
- SSH alias setup
- Ubuntu or RHEL host preparation
- Docker installation and validation
- Rust/toolchain setup
- ETSI GS QKD 014 repository clone or update
- local Docker image build
- Docker network creation or validation
- Docker Compose generation
- KME container lifecycle
- shared PostgreSQL lifecycle
- KME certificate installation
- KME-only restart
- status checks
- validation checks

The only intentional intersection between QKD and KME is:

```text
qkd_orchestrator.py -> certs/* -> kme_orchestrator.py
```

## Architecture Decisions

### One Juniper Device Equals One KME Container

Accepted model:

```text
1 Juniper device = 1 KME container
```

Examples:

```text
2 router pair   -> kme01, kme02
5 router ring   -> kme01..kme05
10 router chain -> kme01..kme10
50 routers      -> kme01..kme50
```

The KME count must be dynamic. No static compose file with a fixed number of KME services should be maintained by hand.

---

## Shared PostgreSQL Model

Accepted model:

```text
1 shared PostgreSQL container for all KME containers
```

Service name:

```text
qkd-postgres
```

Container name pattern:

```text
{owner}-qkd-postgres
```

Example:

```text
aterren-qkd-postgres
```

Deprecated old models:

```text
postgres-kme
postgres-kme1
postgres-kme2
```

Those belong to old lab designs and are not the new default.

---

## Docker Image Naming

Accepted image:

```text
etsi-kme:local
```

Rejected image:

```text
aterren-etsi-kme:local
```

Reason:

- Docker image is generic.
- User identity belongs to container names, not image names.

---

## Container Naming

Service names are generic:

```text

...
```

Container names include owner prefix:

```text
{owner}-qkd-postgres
{owner}-kme01
{owner}-kme02
{owner}-kme03
...
```

Example for owner `aterren`:

```text
aterren-qkd-postgres

aterren-kme02
```

## Network Model

Network driver is configurable.

Supported:

```text

```

Default for the current lab:

```text

```

Current lab network values:

```text
Subnet        : 100.100.100.0/24
Gateway       : 100.100.100.50
KME host IP   : 100.100.100.254
```

## Docker Compose Generation

The compose must be generated dynamically.

Do not maintain a static compose file with fixed `kme01..kme11` entries.

The generator must use:

```text
config/kme/lab.yaml
config/runtime/devices.yaml
```

The count is derived from runtime devices unless explicitly overridden.

Generated file:

```text
<remote ETSI repo>/docker-compose-kme.yml
```

The generated compose includes:

```text
qkd-postgres
kme01..kmeNN
```

---

## PostgreSQL Init

The postgres service must keep:

```yaml
volumes:
  - ./db-init:/docker-entrypoint-initdb.d
```

Decision:

```text
db-init command is not part of the normal lifecycle
```

Reason:

PostgreSQL executes files in `/docker-entrypoint-initdb.d` only during first database initialization.

Normal lifecycle does not need a separate `db-init` command.

A future maintenance command can be added later for schema repair or migration.

---

## Restart Policy

Normal restart must restart KME containers only.

Allowed:

```bash
docker restart aterren-kme01 aterren-kme02
```

Forbidden during normal restart:

```bash
docker restart aterren-qkd-postgres
docker compose down -v
docker compose up -d
```

Reason:

- database must remain persistent
- certificate refresh only requires KME restart
- docker network and volumes must not be recreated during normal maintenance

## Bootstrap Scope

Bootstrap is now limited to:

```text
SSH key generation
public key installation
local SSH config update
passwordless SSH validation
remote OS detection
remote workspace creation
state writing
```

Bootstrap does not clone the ETSI repository anymore.

Repository clone/update moved to:

```text
build_env.py
```

Final bootstrap command:

```bash
python3 -m lib.kme.bootstrap --config config/kme/lab.yaml --dry-run
```

Important:

Run bootstrap as module, not as direct file path:

```bash
python3 -m lib.kme.bootstrap
```

Do not run:

```bash
python3 lib/kme/bootstrap.py
```

Reason:

Direct file execution breaks package imports such as:

```python
from lib.kme.state import save_state
```
